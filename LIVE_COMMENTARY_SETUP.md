# Live Voice Commentary

This implementation adds free browser-to-browser audio commentary with PeerJS and WebRTC.

## Files

- `admin/start_match/score_controller.php`
  - Admin-only Start/Stop commentary controls.
  - Captures microphone audio only.
  - Calls live viewer PeerJS IDs with WebRTC audio.
- `live_stream/live_match.php`
  - Viewer commentary panel.
  - Enable Commentary button for browser autoplay rules.
  - Mute/unmute control and connection status.
- `live_stream/live_commentary_api.php`
  - Lightweight PHP polling/signaling endpoint.
  - Admin/commentator actions: `start`, `stop`, `viewers`.
  - Viewer actions: `status`, `join`, `heartbeat`, `leave`.
- `includes/live_commentary.php`
  - Token generation, protected runtime state, viewer pruning.
- `storage/live_commentary/`
  - Runtime-only room state and secret files.
  - Protected by `.htaccess`.
  - Ignored by Git except `.htaccess` and `.gitignore`.

## Requirements

- Serve the site over HTTPS in production. WebRTC microphone capture will not work on normal HTTP.
- `localhost` is allowed for local XAMPP testing.
- Browser access to `https://cdn.jsdelivr.net/npm/peerjs@1.5.4/dist/peerjs.min.js`.
- Writable `storage/live_commentary/` directory on shared hosting.

## How It Works

1. A viewer opens `live_stream/live_match.php?id=MATCH_ID`.
2. The page receives a hidden HMAC commentary token generated by PHP.
3. The viewer taps `Enable Commentary`, creates a PeerJS receiver ID, and registers it through `live_commentary_api.php`.
4. The admin opens `admin/start_match/score_controller.php?id=MATCH_ID` and clicks `Start Live Commentary`.
5. The admin browser captures microphone audio and polls the PHP endpoint for active viewer peer IDs.
6. The admin browser starts WebRTC audio-only calls to viewers.
7. Viewers receive the live audio stream in the page audio element.
8. No audio files and no database rows are created.

## Security

- Starting, stopping, and fetching viewer peer IDs require session role `admin` or `commentator`.
- The admin broadcaster PeerJS ID is not exposed to viewers.
- Viewer API calls require the server-generated match token.
- Runtime room IDs are random per start.
- Runtime state is stored outside normal app data in `storage/live_commentary/` and blocked by `.htaccess`.

## Audio Settings

The broadcaster requests:

- `echoCancellation: true`
- `noiseSuppression: true`
- `autoGainControl: true`
- mono audio
- 48 kHz input
- Opus SDP preference with low bitrate, DTX, and in-band FEC

## Android WebView

For an Android Studio WebView APK, enable microphone permission and media playback:

```java
webView.getSettings().setJavaScriptEnabled(true);
webView.getSettings().setMediaPlaybackRequiresUserGesture(false);
```

Also request and grant `android.permission.RECORD_AUDIO` in the app. The page still keeps the `Enable Commentary` button because Chrome and WebView autoplay policies vary by Android version.

## Operational Notes

- Peer-to-peer audio means admin upload bandwidth grows with viewer count.
- Free WebRTC without a TURN server can fail on restrictive mobile networks or corporate Wi-Fi.
- The implementation uses public STUN only, so it stays free and shared-hosting compatible.
- For large audiences, use a proper SFU/streaming server in the future.

## Debugging

Open the browser console on both pages and filter for:

- `[LiveCommentary][host]`
- `[LiveCommentary][viewer]`

The logs show microphone capture, PeerJS open/error/disconnect events, room ID sync, viewer heartbeats, outgoing calls, incoming streams, ICE/WebRTC connection state, playback retries, and cleanup.
